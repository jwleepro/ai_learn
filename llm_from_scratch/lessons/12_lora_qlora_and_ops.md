# 12. (선택) LoRA/QLoRA + 메모리/운영 체크리스트

목표:

- “왜 LoRA/QLoRA를 쓰는가”를 VRAM(메모리) 관점에서 이해합니다.
- 파인튜닝을 실제로 돌릴 때 자주 부딪히는 옵션(정밀도/배치/체크포인팅/양자화)을 정리합니다.

> 이 레슨은 구현보다 “용어와 감각”이 목적입니다. 프레임워크(HF/PEFT/TRL 등)에 따라 옵션 이름은 달라질 수 있어요.
>
> 처음 읽을 때는 아래 3줄만 기억해도 충분합니다.
> 1. LoRA = 큰 원본을 거의 그대로 두고 작은 추가분만 학습
> 2. QLoRA = 원본을 더 작게 들고 LoRA를 얹어 학습
> 3. VRAM이 모자라면 보통 `시퀀스 길이`부터 먼저 줄입니다.

처음 읽을 때 잠시 넘겨도 되는 것:

- `rank`, `alpha`, `target modules`의 세부 튜닝
- 4-bit 커널/서빙 엔진 이름

---

## 1) Full fine-tuning이 어려운 이유

큰 모델은 파라미터가 많아서:

- 가중치(파라미터) 자체도 메모리를 먹고
- 학습 중에는 optimizer 상태/gradient/activation까지 같이 들고 있어야 해서
- “그냥 다 업데이트(full fine-tuning)”가 GPU 1장(VRAM 16GB 등)에서 잘 안 돌아갑니다.

그래서 등장한 선택지가 **PEFT(Parameter-Efficient Fine-Tuning)** 입니다.

평범한 말로 바꾸면:

- `activation` = 학습 중 잠깐 저장해 두는 중간 계산값
- `PEFT` = 모델 전체 대신 일부만 가볍게 학습하는 계열

---

## 2) LoRA: “원본은 고정, 작은 추가분만 학습”

**LoRA(Low-Rank Adaptation)** 의 핵심:

- 기존 큰 행렬을 직접 바꾸는 대신
- “작은 랭크(rank) 행렬 2개”를 덧붙여서 업데이트합니다.

비유:

- full fine-tuning = 책 본문 전체를 다시 고쳐 쓰기
- LoRA = 필요한 페이지에 작은 메모지를 붙여 수정하기

실무에서 자주 보는 파라미터/용어:

- **rank(r)**: LoRA가 추가로 배우는 “용량” (작을수록 가볍고, 너무 작으면 성능 한계)
- **alpha**: LoRA 업데이트 스케일(학습 안정/성능에 영향)
- **target modules**: 어떤 레이어(예: attention의 q/k/v/proj)에 LoRA를 붙일지

처음에는 이렇게만 이해해도 충분합니다.

- `rank` = 메모지 크기
- `alpha` = 메모지 효과 강도
- `target modules` = 메모지를 어느 위치에 붙일지

감각:

- “내 태스크가 모델 전체를 바꾸는 게 필요한가?”보다
- “특정 스타일/규칙/도메인 지식을 주입”하는 데 LoRA가 잘 맞는 경우가 많습니다.

---

## 3) QLoRA: “4-bit로 줄여서 더 작은 GPU에서도”

**QLoRA** 는 보통 이렇게 이해하면 충분합니다.

- base model 가중치는 **4-bit로 양자화(quantize)** 해서 VRAM을 크게 줄이고
- 학습은 LoRA 쪽 파라미터 위주로 진행합니다.

비유:

- LoRA = 원본 책은 그대로 두고 메모지 추가
- QLoRA = 원본 책도 더 작은 압축판으로 들고 다니면서 메모지 추가

주의:

- 양자화는 “항상 공짜”가 아닙니다(속도/정확도/커널 호환 이슈).
- 학습용 QLoRA와, 배포용 양자화는 목표가 조금 다를 수 있습니다.

---

## 4) VRAM을 좌우하는 5가지 레버(실무 우선순위)

1. **시퀀스 길이(max length / context length)**:
   - 길이가 2배면 activation 메모리는 보통 더 크게 늘어납니다.
2. **배치 크기(batch size)**:
   - 큰 배치는 안정적일 수 있지만 VRAM을 많이 씁니다.
3. **정밀도(precision)**:
   - FP32(무거움) → FP16/BF16(가벼움) → (일부) 8-bit/4-bit
4. **gradient accumulation**:
   - “작은 micro-batch”를 여러 번 누적해서, 큰 배치처럼 학습하는 트릭
5. **gradient checkpointing**:
   - activation을 모두 저장하지 않고, 필요할 때 다시 계산해서 메모리를 아끼는 트릭(대신 느려짐)

처음 실험할 때는 아래 순서로 조절하면 됩니다.

1. `max length` 줄이기
2. `batch size` 줄이기
3. `gradient accumulation`으로 보정하기
4. 그래도 부족하면 LoRA/QLoRA 검토하기

---

## 5) 평가/회귀 테스트: 파인튜닝에서 더 중요해지는 이유

파인튜닝은 바뀌는 게 많습니다.

- 데이터가 조금만 바뀌어도 결과가 바뀌고
- 프롬프트 템플릿이 바뀌어도 결과가 바뀌고
- 양자화/서빙 엔진이 바뀌어도 결과가 바뀝니다.

그래서 최소한 아래는 고정하는 게 좋습니다.

- 고정된 **테스트셋**(10~100개라도 OK)
- **형식 준수 검사**(JSON 파싱/스키마/정규식)
- “이전 버전 대비 나빠지면 경고”하는 **회귀(regression) 체크**

---

## 6) 배포 쪽 용어: quantization vs distillation

- **quantization(양자화)**: 가중치/연산을 더 낮은 비트로 바꿔서 빠르고 싸게 서빙
- **distillation(지식 증류)**: 큰 모델의 출력을 “정답”으로 삼아 작은 모델을 학습시키는 방식

둘 다 목표는 “비용/지연 감소”지만 접근이 다릅니다.

---

## 7) 16GB에서 자주 겪는 문제 체크리스트

- OOM이 나면:
  - max length부터 줄이고
  - micro-batch를 줄이고 + gradient accumulation으로 보정하고
  - FP16/BF16, checkpointing, LoRA/QLoRA 순으로 적용
- 성능이 안 오르면:
  - 데이터/정답 품질(ground truth)을 먼저 의심
  - “형식 준수”와 “내용 품질”을 분리해 평가
  - 학습률/스텝/early stopping(과적합)을 점검

한 줄 요약:

- 메모리 문제는 보통 설정 문제부터 줄이고
- 품질 문제는 보통 데이터 문제부터 의심합니다.
